23일차: 파드의 보호막, 보안 컨텍스트(Security Context)
1. 보안 컨텍스트란 무엇인가요?
파드나 컨테이너가 실행될 때, **"이 컨테이너는 어떤 권한까지 가질 수 있는가?"**를 아주 세밀하게 정의하는 설정값입니다.
리눅스 시스템 보안의 핵심인
특권 제한
을 쿠버네티스 설정 파일(YAML) 수준에서 구현하는 것입니다.
2. 보안 전문가를 위한 '필수 체크리스트' 3가지
보안 15년 차이신 사용자님께서 개발팀의 설정 파일을 검수할 때, 다음 3가지는 꼭 확인하셔야 합니다.
allowPrivilegeEscalation: false
컨테이너 내부의 프로세스가 부모 프로세스보다 더 높은 권한을 가질 수 없도록 차단합니다. (권한 상승 공격 방어)
runAsNonRoot: true
컨테이너가 절대로
root
사용자로 실행되지 않도록 강제합니다. 해커가 침투해도 시스템 전체를 장악하기 어렵게 만듭니다.
readOnlyRootFilesystem: true
컨테이너 내부의 파일 시스템을 '읽기 전용'으로 만듭니다. 해커가 악성 파일을 내려받거나 로그를 조작하는 것을 원천 봉쇄합니다.
3. 왜 이게 중요한가요?
보통 컨테이너는 호스트 OS의 커널을 공유합니다. 만약 보안 컨텍스트 설정이 허술하면, 해커가 컨테이너를 뚫고 나와
실제 서버(호스트)까지 장악하는 '컨테이너 탈출(Container Escape)'
사고가 발생할 수 있습니다.
4. 쉬운 비유
컨테이너:
호텔의
'객실'
보안 컨텍스트:
객실 투숙객에게 주는
'이용 규칙'
"객실 안의 가구는 옮길 수 없습니다(ReadOnly)"
"투숙객은 절대 직원의 마스터키를 가질 수 없습니다(Non-Root)"
"객실 밖 복도로 나가는 문은 안쪽에서 잠깁니다(No Privilege Escalation)"
💡 오늘의 요약
"보안 컨텍스트는 컨테이너에게 꼭 필요한 최소한의 권한만 주는 '수동 방화벽'이다. 이 설정을 꼼꼼히 하는 것만으로도 대부분의 치명적인 컨테이너 해킹 사고를 예방할 수 있다!"
실무 활용 팁:
지난번에 배운
Kyverno(키베르노)
정책을 사용하면, "보안 컨텍스트 설정이 누락된 파드는 아예 배포되지 않게" 자동화할 수 있습니다. 이것이 진정한 **'보안의 자동화(DevSecOps)'**입니다.
