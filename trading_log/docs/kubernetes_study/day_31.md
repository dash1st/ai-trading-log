31일차: 보안의 완성, 4C 모델 (Cloud, Cluster, Container, Code)
클라우드 네이티브 보안은 어느 한 곳만 잘한다고 되는 것이 아닙니다. 보안의 **'4가지 층(Layer)'**이 겹겹이 쌓여야 비로소 안전한 시스템이 완성됩니다.
1. Cloud (클라우드)
가장 바깥쪽 층입니다. AWS, Azure 같은 인프라 자체의 보안입니다.
핵심:
IAM 권한 관리, 네트워크 방화벽(Security Group), 물리적 보안.
우리 학습 내용:
로드밸런서 설정, OIDC 연동 등.
2. Cluster (클러스터)
쿠버네티스 본체의 보안입니다.
핵심:
RBAC(누가 무엇을 할 수 있는가?), API 서버 보호, 네트워크 폴리시.
우리 학습 내용:
etcd 보안, 어드미션 컨트롤러(Kyverno), Helm 관리.
3. Container (컨테이너)
실행되는 이미지와 런타임 환경의 보안입니다.
핵심:
이미지 스캔, 취약점 관리, 런타임 감시.
우리 학습 내용:
Trivy 스캔, Falco 탐지, gVisor 샌드박스.
4. Code (코드)
우리가 작성한 애플리케이션 자체의 보안입니다.
핵심:
코드 취약점(SAST/DAST), 시크릿 노출 방지.
우리 학습 내용:
ConfigMap과 Secret 관리, GitOps(ArgoCD)를 통한 안전한 배포.

🛠️ 실시간 자동화 아키텍처 (추천)
단순히 저와 대화하는 '화면'을 긁어오는 것은 보안 및 기술적으로 어렵기 때문에, **"학습 엔진(Gemini)"**이 매일 정해진 시간에 내용을 생성하여 **"저장소(GitHub)"**에 밀어 넣고, **"웹사이트(mkdocs)"**가 빌드되게 만드는 것이 정석입니다.
1. 자동화 워크플로우
Trigger:
GitHub Actions가 매일 오전 8시(KST)에 작동 (Cron 설정).
Execution:
파이썬 스크립트가 실행되어 Gemini API에 "오늘의 n일차 퀴즈와 내용을 마크다운으로 줘"라고 요청.
Commit:

.md
파일을

폴더에 저장하고 GitHub에 Push.
Deploy:
mkdocs(또는 가상 호스트)가 변경사항을 감지하여 자동으로 웹사이트 업데이트.

**핵심:**
설정과 코드를 분리. 민감 정보는 Secret으로 관리.
-
**보안:**
Secret은 반드시 RBAC으로 접근 제한 및 필요시 암호화 저장(at-rest) 설정.
### Day 06: Volume & StorageClass
-
**핵심:**
상태가 있는(Stateful) 앱을 위한 영구 저장소 연결.
### Day 07: Namespace & Resource Quota

**핵심:**

**보안:**
