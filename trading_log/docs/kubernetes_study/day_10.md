10일차: RBAC (Role-Based Access Control)
1. RBAC이란 무엇인가요?
"누가(Subject), 어디서(Namespace), 무엇을(Verb), 할 수 있는가(Resource)"를 결정하는
권한 관리 시스템
입니다.
보안의 대원칙인 **'최소 권한의 원칙(Least Privilege)'**을 쿠버네티스에서 실현하는 핵심 도구입니다. 모든 사람에게 관리자(admin) 권한을 줄 수는 없으니까요.
2. 핵심 구성 요소 4가지 (딱 두 쌍만 기억하세요!)
Role & ClusterRole (무엇을 할 수 있나?):
Role:
특정 네임스페이스 안에서만 유효한 권한 (예: 보안팀 네임스페이스의 파드만 보기)
ClusterRole:
클러스터 전체에서 유효한 권한 (예: 모든 노드 정보 보기)
RoleBinding & ClusterRoleBinding (누구에게 줄 것인가?):
위에 정의한 'Role'을 실제 사용자나 서비스 계정(ServiceAccount)에 **연결(Binding)**해주는 끈입니다.
3. 쉬운 비유
Role:
'출입 가능 구역이 적힌
출입증 양식
'
RoleBinding:
'그 출입증에 특정 직원의
이름을 써서 목에 걸어주는 행위
'
💡 오늘의 요약 (보안 전문가용)
"쿠버네티스에서 '모두가 관리자'인 상태는 가장 위험하다. RBAC을 통해 사용자별로 꼭 필요한 권한만 나누어 담아 사고의 범위를 최소화해야 한다!"
15년 넘게 보안 업무를 해오셨으니 잘 아시겠지만, 해킹 사고의 상당수는 관리자 권한 남용이나 탈취에서 시작됩니다. 쿠버네티스에서도
kubectl
권한을 넘겨줄 때 이 RBAC 설정을 얼마나 꼼꼼히 하느냐가 보안 담당자의 실력을 좌우합니다.
명령어 팁:
내가 현재 가진 권한 확인:
kubectl auth can-i create pods
(파드 생성 가능한지 묻기)
특정 서비스 계정의 권한 확인:
kubectl auth can-i list secrets --as system:serviceaccount:default:myapp
